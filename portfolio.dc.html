Project 01 · 주요 기능
핵심 기능 상세
http://localhost/Cinema_Talk/movieRecommend.do
01
선호 장르 기반 영화 추천
로그인 여부 → 선호 장르 설정 여부에 따라 3단계로 분기합니다. 비로그인이면 랜덤, 선호 장르 미설정이면 랜덤, 선호 장르가 있으면 해당 장르 영화를 DBMS_RANDOM으로 셔플해 매번 다른 추천 결과를 제공합니다.
02
인기 영화 추천
평점 20% + 회원 추천수 80% 가중치로 점수를 계산해 상위 60개를 추린 뒤 DBMS_RANDOM으로 셔플합니다. 단순 정렬이 아닌 정규화 스코어링으로 인기 편향 없이 다양한 영화를 노출합니다
03
장르별 섹션 그룹핑
DB에서 장르별로 인기 상위 100개 → 랜덤 20개를 추린 뒤, 동일 영화 중복 노출을 ROW_NUMBER() OVER (PARTITION BY movieId)로 제거합니다. Java에서는 LinkedHashMap의 computeIfAbsent로 장르 순서를 유지하며 섹션 단위로 그룹핑합니다.

무엇을 하는 코드인가
모든 .do 요청을 하나의 서블릿이 받아, Properties 파일에서 URL에 해당하는 클래스명을 찾고 Reflection으로 인스턴스를 생성해 실행합니다. 결과에 따라 redirect 또는 forward로 응답합니다.
핵심 포인트
Class.forName()+getDeclaredConstructor().newInstance()로 컨트롤러를 동적 생성해, 새 기능 추가 시 Java 코드 수정 없이 properties 파일 한 줄만 추가하면 됩니다. Spring의 DispatcherServlet이 내부적으로 하는 일을 직접 구현한 구조입니다.

위젯 배치 정보를 JSON CLOB으로 저장해 직원마다 독립적인 레이아웃을 유지합니다. 위젯 추가 시 테이블 스키마 변경 없이 JSON 구조만 확장하면 되어 유연한 확장이 가능합니다. 직원이 재배치한 위젯은 다음 접속 시 그대로 복원됩니다.
02
위젯별 독립 API 부분 렌더링 최적화
12종 위젯을 각각 독립 엔드포인트로 분리했습니다. 하나의 위젯에 오류가 발생해도 다른 위젯에 영향을 주지 않으며, 프론트에서 병렬 호출로 전체 로딩 시간을 단축합니다. 필요한 위젯 데이터만 선택적으로 갱신할 수 있습니다.
03
역할별 기본 레이아웃 코드 관리
최초 접속 사용자의 기본 레이아웃을 DB가 아닌 Java 코드에서 역할별로 정의했습니다. DB 초기 데이터 없이도 역할에 맞는 위젯 구성이 자동 생성되며, 기본 레이아웃 변경 시 SQL 없이 코드 수정만으로 반영됩니다.
Project 02 · 핵심 코드
코드 하이라이트
StompChannelInterceptor .java
01 @Override
02 public Message<?> preSend(Message<?> message, MessageChannel channel) {
03 StompHeaderAccessor accessor =
04 MessageHeaderAccessor.getAccessor(message, StompHeaderAccessor.class);
05 // 최초 WebSocket 연결(CONNECT) 요청에서만 JWT 인증 수행
06 if (accessor != null && StompCommand.CONNECT.equals(accessor.getCommand())) {
07 String authHeader = accessor.getFirstNativeHeader("Authorization");
08 if (authHeader == null || !authHeader.startsWith("Bearer ")) {
09 throw new CustomException(ErrorCode.WS_UNAUTHORIZED);
10 }
11 String token = authHeader.substring(7);
12 jwtTokenProvider.validateToken(token);
13 // 검증된 토큰에서 사원 ID를 추출해 WebSocket 사용자 정보로 등록
14 Long empId = jwtTokenProvider.getEmpId(token);
15 accessor.setUser(() -> String.valueOf(empId));
16 }
17 return message;
18 }
무엇을 하는 코드인가
WebSocket STOMP 연결(CONNECT) 시점에 Authorization 헤더에서 JWT를 추출해 검증하고, 사원 ID를 Principal로 등록합니다. 이후 SEND·SUBSCRIBE 메시지에서도 별도 인증 없이 사원 ID를 그대로 사용할 수 있습니다.
핵심 포인트
HTTP 기반 Spring Security 필터는 WebSocket 환경에서 동작하지 않아 ChannelInterceptor를 직접 구현했습니다. StompHeaderAccessor.wrap() 대신 MessageHeaderAccessor.getAccessor()를 사용한 이유는, wrap은 새 객체를 만들어 헤더 수정이 원본 메시지에 반영되지 않지만 getAccessor는 기존 객체를 그대로 반환해 setUser() 결과가 실제 메시지에 적용되기 때문입니다.
POINT — HTTP 보안 필터가 닿지 않는 WebSocket 영역에
별도 인증 흐름을 설계했습니다.
Project 02 · 핵심 코드
코드 하이라이트
MessengerEventListener .java
01 @Component
02 @RequiredArgsConstructor
03 public void MessengerEventListener {
04 private final MsngrService msngrService;
05 private final ProjectService projectService;
06
07 @Async
08 @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
09 public void MessengerEventListener(ProjectCreatedEvent event) {
10 try{
11 // 메신저 방 자동 생성
12 Long chtrmId = msngrService.createProjectChtrm(...);
13 // 생성된 방 ID를 프로젝트 테이블에 업데이트
14 projectService.updateProjectChtrmId(event.getProjId, chtrmId);
15 } catch (Exception e) {
16 log.error("메신저 채팅방 자동 생성 실패", e); // 예외(에러) 로그
17 }
18 }
무엇을 하는 코드인가
프로젝트 생성 시 DB 커밋이 완료된 직후, 해당 프로젝트의 메신저 채팅방을 자동으로 생성하고 방 ID를 프로젝트 테이블에 업데이트합니다. 사용자는 별도 설정 없이 프로젝트 생성과 동시에 전용 채팅방에 입장할 수 있습니다.
핵심 포인트
@TransactionalEventListener(AFTER_COMMIT)으로 DB 커밋 완료 후에만 채팅방이 생성되어 트랜잭션 롤백 시 프로젝트 없이 채팅방만 따로 생성되어 남는 문제를 방지했습니다. 프로젝트 서비스는 이벤트를 발행하는 것만 담당하고, 메신저 채팅방 생성 로직은 리스너에 완전히 위임해 도메인 간 결합력을 낮췄습니다.
POINT — DB 커밋 이후 채팅방을 생성하도록 분리해,
프로젝트 생성과 메신저 연동의 데이터 정합성을 확보했습니다.

상세 설명
회의실 예약 등록 시, 같은 회의실의 동일 시간대 중복 예약을 DB 쿼리 레벨에서 차단합니다. 사용자가 회의실과 시간을 선택하면 Controller가 요청을 받아 Service로 전달하고, Service는 Mapper를 통해 countOverlappingRsrv 쿼리를 실행해 같은 회의실(CONF_RM_ID)·같은 시간대에 정상 예약(DEL_YN = 'N')이 존재하는지 확인합니다. 구간 겹침 판정 조건은 BEGIN_DT < 요청종료 AND END_DT > 요청시작으로, 등호를 제외해 딱 붙는 예약(9시~10시 / 10시~11시)은 허용합니다. 수정 요청의 경우 본인 예약(RSRV_ID != #{rsrvId})을 제외해 자기 자신과의 충돌을 방지합니다. 중복이 감지되면 INSERT 없이 즉시 예외를 반환하고, 프론트에 에러 메시지를 표시합니다.

Project 02 · Troubleshooting
트러블슈팅
CASE 01
문제
화상회의와 회의실 예약을 별도 기능으로 설계해 회의록/참석자/결재가 화상회의에 종속됨
발견 계기
기능 구현을 마친 후 현직 개발자 지인에게 실제 업무 프로세스를 들었습니다. 오프라인 회의에서도 회의록을 작성해 결재받아야 하고, 화상회의도 회의실에서 진행하는 경우가 많다는 사실을 그제야 인식했습니다. 구현 기술(LiveKit)을 기준으로 설계하다 보니 "회의"라는 상위 도메인 개념 자체가 빠진 것이 근본 원인이었습니다.
해결
TB_MTNG를 상위 도메인으로 신설해 온라인/오프라인 회의를 통합했습니다.
  - 회의 통합: MTNG_TYPE_CD로 온라인/오프라인/혼합 구분
  - 회의록 재구성: TB_VIDEO_MOM → TB_MTNG_MOM, MTNG_ID 기준 관리
  - 참석자 분리: 참석자 명단과 입퇴장 로그 분리
  - AI 회의록 처리: @Async + AFTER_COMMIT으로 별도 처리해 응답 지연 방지
배운 점
실제 업무 프로세스 없이 구현 기술 기준으로 설계하면 기술적으로는 작동하지만 현실과
맞지 않는 구조가 나오게 됩니다. "이 기능이 다른 진행 방식에서도 재사용될 수 있는가?"를 설계 단계에서 먼저 질문하는 습관을 가지게 되었습니다.
Project 02 · Troubleshooting
트러블슈팅
CASE 02
문제
통합 검색창 타이핑 시 실시간 자모음 합성으로 인해 무분별한 중복 API 요청이 발생함.
단어 하나 입력당 10회 이상의 대형 쿼리가 유입되어 서버 및 DB 과부하를 유발함.
발견 계기
기능 구현 후 프론트엔드 연동 테스트 과정에서 입력 속도에 비례해 화면이 미세하게 버벅거리는 현상을 인지하였습니다. 백엔드 콘솔에 동일한 통합 검색 SQL 로그가 순간적으로 폭발하듯 찍히는 것을 보고 성능 병목 위험을 확인하였습니다.
해결
입력 이벤트 지연을 위한 250ms 디바운싱(Debouncing) 기법을 컴포넌트에 도입.
  - 타이머 제어: useEffect와 setTimeout을 활용해 타자 입력 중인 중간 요청을
물리적으로 차단
  - 부하 절감: 사용자 입력이 완전히 끝난 시점에만 최종적으로 단 1회 요청하도록 제어
배운 점
크롬 개발자 도구(Network 탭) 실측 결과 API 호출 횟수가 80% 이상 절감됨을 확인하였습니다. 대량의 데이터를 다루는 LIKE 검색 특성상 연쇄적인 Full Table Scan 오버헤드를 사전에 차단하는 것이 중요함을 깨달았습니다. 단순 기능 구현을 넘어 서버와 DB 리소스를 고려하는 설계 시야를 갖추게 되었습니다.
